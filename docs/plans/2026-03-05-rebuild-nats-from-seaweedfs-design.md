# Rebuild NATS Trip Data from SeaweedFS

## Problem

The NATS JetStream `trips` stream had `max_age: 30d`. The last data was published on 2026-01-07, so all 21,864 messages expired and were purged. The trips API replays this stream on startup to build its in-memory cache — with an empty stream, `api.jomcgi.dev/trips/api/points` returns `{"points":[],"total":0}` and `trips.jomcgi.dev` shows a blank page.

Images (4,878 JPEGs) are intact in the SeaweedFS `trips` S3 bucket. EXIF data embedded in the images contains GPS coordinates, timestamps, and camera settings.

## Solution

Add a `rebuild` command to `scripts/publish-trip-images/main.py` that recovers trip metadata from SeaweedFS images and republishes to NATS. Also fix the stream retention to prevent recurrence.

## Design

### Command Interface

```
publish-trip-images rebuild [OPTIONS]

Options:
  --bucket TEXT         S3 bucket name [default: trips]
  --batch-size INT      Images per download batch [default: 80]
  --concurrency INT     Parallel downloads per batch [default: 10]
  --dry-run / -n        Extract and report only, don't publish
  --source TEXT         Image source tag [default: gopro]
  --db PATH             Path to publish_queue.db
  --fix-retention       Update NATS stream max_age to unlimited [default: True]
```

### Flow

```
1. Connect to NATS + S3
2. If --fix-retention: update stream max_age=0 (unlimited)
3. List all S3 object keys (paginated via list_objects_v2, keys only in memory)
4. For each batch of 80 images:
   a. Download images in parallel to temp dir (10 workers)
   b. Extract EXIF (GPS, timestamp, optics) via existing extract_exif()
   c. Skip images without valid GPS coordinates
   d. Add to publish_queue.db via UploadQueue.add() (mark completed)
   e. Delete temp directory contents
5. Fetch elevation for all collected GPS points via ElevationClient:
   - SQLite cache at scripts/elevation/elevation_cache.db
   - NRCan CDEM API, rate-limited (50 concurrent, 0.5s between batches)
6. Publish all points to NATS via publish_to_nats() (with elevation)
7. Print final stats
```

### Disk Usage

Batch size of 80 images × ~5.5MB avg = ~440MB peak. Temp directory is wiped between batches. Total disk budget stays under 500MB.

### NATS Message Shape

```json
{
  "id": "abc123def456",
  "lat": 49.2827,
  "lng": -123.1207,
  "timestamp": "2025-12-28T14:30:00",
  "image": "img_abc123def456.jpg",
  "source": "gopro",
  "tags": [],
  "elevation": 125.5,
  "light_value": 8.6,
  "iso": 393,
  "shutter_speed": "1/240",
  "aperture": 2.5,
  "focal_length_35mm": 16
}
```

### Reused Components

- `extract_exif()` — GPS, timestamp, optics from EXIF
- `publish_to_nats()` — builds and publishes trip point message
- `UploadQueue` — SQLite queue DB for tracking
- `OpticsCache` — SQLite cache for optics data
- `ElevationClient` — async NRCan API client with SQLite cache
- `get_s3_client()` — boto3 S3 client for SeaweedFS
- Rich progress bars — consistent with existing commands

### New Dependency

`publish-trip-images` BUILD adds dep on `//scripts/elevation` for `ElevationClient`.

### NATS Stream Fix

Before publishing, update the stream config:

```python
await js.update_stream(name="trips", max_age=0)  # unlimited
```

This removes the 30-day TTL while preserving the existing `max_bytes: 1GiB` limit (sufficient for millions of ~500 byte point messages).

### What's NOT Recovered

- **Custom tags** — original tags like "hotspring", "wildlife" are lost. All rebuilt points get empty tags `[]`. User can re-tag via the publish script later.
- **Gap points** — synthetic route-fill points (source="gap", no image) were generated by the ingest pipeline and cannot be recovered from images alone.

### Idempotency

- `UploadQueue.add()` uses `dest_key` as unique constraint — reruns skip already-processed images
- NATS messages with the same `id` overwrite in the API's in-memory cache (last-writer-wins)
- `ElevationClient` caches results in SQLite — re-fetches are instant
