# Repo Reorganization Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Restructure the homelab monorepo from layer-first (services/, charts/, overlays/) to domain-first (projects/) with colocated deploy directories and no environment overlays.

**Architecture:** Incremental migration -- one PR per domain/concern. Each PR moves source + chart + overlay into the new `projects/` structure, updates ArgoCD Application paths, and fixes Bazel references. Auto-generated BUILD files (initially `images/BUILD` and `websites/BUILD`, moving to `bazel/images/BUILD` and `projects/websites/BUILD` after Tasks 1 and 5) are regenerated by running `format`.

**Inner convention:** Every subproject uses standardized subdirectories: `backend/`, `frontend/`, `ingest/`, `deploy/`, `tools/`. Use only what applies.

**Tech Stack:** git mv, Bazel (BUILD files, bzl imports), Helm (Chart.yaml, values.yaml), ArgoCD (Application manifests), Kustomize (kustomization.yaml)

---

## Migration Strategy

Each task follows the same pattern:

1. `git mv` source files to new location
2. Update Bazel `BUILD` files ÔÇö fix `load()` paths, `deps`, `visibility`
3. Move chart into `deploy/` within the subproject
4. Merge overlay values into `deploy/values.yaml`
5. Update ArgoCD `application.yaml` path to point to new `deploy/`
6. Update `clusters/homelab/kustomization.yaml` to reference new path
7. Run `format` to regenerate auto-generated BUILD files
8. Commit

**Important:** ArgoCD Applications reference `path: charts/<name>` in the git repo. When we move the chart into `deploy/`, the `path:` must change to the new location (e.g. `projects/agent-platform/orchestrator/deploy`). ArgoCD will auto-sync once the commit lands on main.

**Values merge strategy:** Currently, ArgoCD applies both `charts/<svc>/values.yaml` (chart defaults) and `overlays/<env>/<svc>/values.yaml` (overrides). When collapsing into `deploy/`, merge both into a single `deploy/values.yaml` -- overlay values take precedence. Also update `helm.valueFiles` in `application.yaml` to reference the colocated `values.yaml` (removing any `../../overlays/...` paths).

**Order matters:** Tasks 1-2 (bazel + docs) have no cluster impact and can land first. Tasks 3-8 each move a domain and must update the kustomization atomically.
