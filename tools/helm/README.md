# Helm Bazel Rules and Gazelle Extension

This directory contains Bazel rules and a Gazelle extension for working with Helm charts and ArgoCD applications.

## Overview

The tools in this directory enable:
1. **Rendering Helm charts** - See what ArgoCD will deploy without applying
2. **Diffing against cluster** - Compare rendered manifests with deployed state
3. **Auto-generating BUILD files** - Gazelle discovers ArgoCD applications and creates targets

## Architecture

```
tools/helm/
├── defs.bzl                 # Public API (load this in BUILD files)
├── private/
│   ├── render.bzl           # helm_render rule implementation
│   └── diff.bzl             # helm_diff_script rule implementation
├── language.go              # Gazelle language interface
├── config.go                # Gazelle directive parsing
├── generate.go              # BUILD file generation logic
├── BUILD                    # Builds the Gazelle extension
├── go.mod                   # Go module definition
├── README.md                # This file
└── TEST.md                  # Testing instructions
```

## Usage

### Manual Rule Usage

Load the rules in a BUILD file:

```starlark
load("//tools/helm:defs.bzl", "helm_render", "helm_diff_script")

helm_render(
    name = "render",
    chart = "//charts/n8n:Chart.yaml",
    release_name = "n8n",
    namespace = "n8n",
    values = [
        "//charts/n8n:values.yaml",
        "values.yaml",
    ],
)

helm_diff_script(
    name = "diff",
    rendered = ":render",
    namespace = "n8n",
    kubectl_context = "homelab",
)
```

Run with:
```bash
bazel run //overlays/prod/n8n:render  # Render manifests
bazel run //overlays/prod/n8n:diff    # Diff against cluster
```

### Automatic Generation with Gazelle

The Gazelle extension automatically discovers ArgoCD `application.yaml` files and generates BUILD files.

1. **Run Gazelle:**
   ```bash
   bazel run //:gazelle
   ```

2. **Gazelle discovers** all `application.yaml` files in `overlays/` and generates BUILD files with:
   - `helm_render` rule (always)
   - `helm_diff_script` rule (if enabled)

3. **Control generation** with directives in kustomization.yaml:
   ```yaml
   # gazelle:argocd_generate_diff true
   # gazelle:kubectl_context homelab
   ```

## Rules Reference

### helm_render

Renders a Helm chart to Kubernetes manifests.

**Attributes:**
- `chart` (label) - Path to Chart.yaml file
- `release_name` (string) - Helm release name
- `namespace` (string) - Kubernetes namespace
- `values` (list of labels) - Values files to pass to helm template
- `helm_flags` (list of strings) - Additional helm template flags

**Example:**
```starlark
helm_render(
    name = "render",
    chart = "//charts/n8n:Chart.yaml",
    release_name = "n8n",
    namespace = "n8n",
    values = [
        "//charts/n8n:values.yaml",
        "values.yaml",
    ],
)
```

### helm_diff_script

Creates a script that diffs rendered manifests against cluster state.

**Attributes:**
- `rendered` (label) - Output of helm_render rule
- `namespace` (string) - Kubernetes namespace
- `kubectl_context` (string) - kubectl context (default: "current")

**Example:**
```starlark
helm_diff_script(
    name = "diff",
    rendered = ":render",
    namespace = "n8n",
    kubectl_context = "homelab",
)
```

## Gazelle Extension

The Gazelle extension parses ArgoCD Application manifests and generates BUILD rules.

### Directives

Control BUILD generation with these directives:

| Directive | Values | Description |
|-----------|--------|-------------|
| `argocd` | `enabled`, `disabled` | Enable/disable BUILD generation |
| `argocd_enabled` | (none) | Enable BUILD generation |
| `argocd_generate_diff` | `true`, `false` | Generate diff rules |
| `kubectl_context` | context name | Set kubectl context for diff |

### How It Works

1. **Discovery**: Gazelle scans directories for `application.yaml` files
2. **Parsing**: Reads Application spec to extract:
   - Chart path (`spec.source.path`)
   - Release name (`spec.source.helm.releaseName`)
   - Namespace (`spec.destination.namespace`)
   - Values files (`spec.source.helm.valueFiles`)
3. **Generation**: Creates BUILD file with appropriate rules

### Example Generated BUILD File

For an ArgoCD Application at `overlays/prod/n8n/application.yaml`:

```starlark
# Generated by gazelle - DO NOT EDIT
load("//tools/helm:defs.bzl", "helm_render", "helm_diff_script")

helm_render(
    name = "render",
    chart = "//charts/n8n:Chart.yaml",
    release_name = "n8n",
    namespace = "n8n",
    values = [
        "//charts/n8n:values.yaml",
        "values.yaml",
    ],
)

helm_diff_script(
    name = "diff",
    rendered = ":render",
    namespace = "n8n",
)
```

## Implementation Notes

### Why Bazel Rules?

Bazel provides:
1. **Hermetic builds** - Helm version is pinned in multitool
2. **Dependency tracking** - Rebuilds when values change
3. **Caching** - Fast re-renders
4. **Composable** - Easy to extend with new rules

### Why Gazelle Extension?

Manual BUILD files are error-prone. Gazelle:
1. **Reads the source of truth** - ArgoCD Application manifests
2. **Stays in sync** - Re-run to update BUILD files
3. **Reduces boilerplate** - Same pattern for all services

### Multitool Integration

Helm is provided via rules_multitool for:
- **Version pinning** - Same helm version everywhere
- **Cross-platform** - Works on Linux, macOS
- **Hermetic** - No system helm required
